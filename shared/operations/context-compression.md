# Context Compression - 上下文压缩策略

_借鉴 Claude Code 的5种上下文压缩策略_
_Created: 2026-04-15_

---

## 📖 问题背景

Claude Code 源码显示，即使他们也在应对上下文溢出问题：
- 5种不同的压缩策略
- 14种缓存失效向量
- 模型需要平衡上下文大小 vs 响应质量

---

## 🎯 目标

当 OpenClaw 上下文过长时，按优先级压缩：

| 压缩级别 | 触发条件 | 保留内容 |
|----------|----------|----------|
| L1 | 上下文>50% | 低相关信息过滤 |
| L2 | 上下文>70% | 历史摘要 |
| L3 | 上下文>85% | 核心任务+关键记忆 |
| L4 | 上下文>95% | 只保留MEMORY.md |

---

## 🗜️ 压缩策略

### L1: 低相关信息过滤

**触发**: 上下文 >50%

```markdown
保留:
- 最近10条对话
- 活跃任务相关
- MEMORY.md 完整

移除:
- 超过30天的历史记录
- 与当前任务无关的闲聊
- 重复的确认信息
```

---

### L2: 历史摘要

**触发**: 上下文 >70%

```markdown
对每段历史对话生成摘要:

[2026-04-10 ~ 2026-04-12] 
- 服务器: 马来西亚染发膏升级Django 4.2
- 日报: 完成7天东南亚电商资讯收集
- 任务: 印尼地坪漆SSL证书续期
```

---

### L3: 任务聚焦

**触发**: 上下文 >85%

```markdown
只保留:
1. MEMORY.md（精简版）
2. 当前任务brief
3. 最近3次交互
4. 必要的技术上下文

格式: 任务聚焦模式
- 丢弃所有中间步骤
- 只保留最终结果
- 标记需要继续的位置
```

---

### L4: 最小化上下文

**触发**: 上下文 >95%

```markdown
只保留:
- MEMORY.md（核心部分）
- 当前紧急任务
- 下一步行动指令
```

## 🔍 关键词过滤

### 应该过滤的

| 类型 | 示例 | 过滤原因 |
|------|------|----------|
| 重复确认 | "好的", "收到", "OK" | 无信息量 |
| 调试日志 | "开始执行...", "完成" | 非关键 |
| 临时文件 | .tmp, .log | 无持久价值 |
| 过期提醒 | "记得xxx" (已过30天) | 无效信息 |

### 应该保留的

| 类型 | 示例 | 保留原因 |
|------|------|----------|
| 决策记录 | "决定用方案A" | 重要结论 |
| 任务结果 | "服务器已升级" | 成果 |
| 知识更新 | "坤哥喜欢简洁风格" | 偏好 |
| 错误教训 | "不要乱汇报" | 重要规则 |

## 📊 压缩触发检测

### 检测方式

```bash
# 每10次交互检查一次上下文大小
# 或每次HEARTBEAT时检查
```

### 检测脚本思路

```python
# 伪代码
if context_length > THRESHOLD_L1:
    apply_compression(L1)
elif context_length > THRESHOLD_L2:
    apply_compression(L2)
elif context_length > THRESHOLD_L3:
    apply_compression(L3)
elif context_length > THRESHOLD_L4:
    apply_compression(L4)
```

## ⚠️ 注意事项

1. **不丢失关键信息** - 决策、规则、偏好必须保留
2. **保留追溯能力** - 压缩后仍能回答"之前做了什么"
3. **透明处理** - 用户应知道何时压缩了上下文
4. **可回滚** - 保存原始日志，支持必要时恢复
